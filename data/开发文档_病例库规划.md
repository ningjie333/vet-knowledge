## 二、开发规划

### Phase 1：病例浏览（核心 MVP）

**目标**：用户可以浏览病例列表、查看完整病例详情

#### 1.1 数据层完善

- 修复 TypeScript `Case` 接口，补全 `lab_results` / `imaging` / `learning_points`
- 新增后端命令 `get_case_diseases(case_id)` → 返回关联疾病列表
- 全文搜索表新增 `cases_fts`（title + chief_complaint + diagnosis）

#### 1.2 CaseLibrary.vue — 病例列表页

```
布局：顶部筛选栏 + 病例卡片网格

筛选控件：
- 物种下拉 [全部/犬/猫]
- 难度下拉 [全部/Basic/Intermediate/Advanced]
- 搜索框（调用全文搜索）

卡片信息：
- 标题
- 物种 | 品种 | 年龄 | 体重
- 难度徽章（绿/橙/红）
- 主诉（截断 2 行）
- 关联疾病名（可点击跳转）
```

#### 1.3 CaseDetail.vue — 病例详情页（新建）

```
路由：/cases/:id

布局：左侧主内容 + 右侧信息面板

主内容（按节展开）：
- 主诉 → 病史 → 体格检查 → 实验室检查 → 影像学 → 诊断 → 治疗 → 转归 → 学习要点
- 每节可折叠（collapse）
- 诊断节高亮显示

右侧面板：
- 动物信息卡片（物种/品种/年龄/体重）
- 难度标签
- 关联疾病链接（→ DiseaseDetail）
- 相关病例推荐（同疾病或同系统）
```

**预估**：前端 2 个页面 + 后端 1 条新命令 + 1 个搜索表迁移

---

### Phase 2：病例学习（推理训练）

**目标**：基于病例的逐步推理训练，模拟临床诊断思维

#### 2.1 新增 CaseStudy.vue — 逐步推理模式

```
路由：/cases/:id/study

交互流程：
1. 只显示"主诉" → 用户思考"下一步该做什么？"
2. 点击揭示"病史" → 继续思考
3. 逐步揭示：体格检查 → 实验室 → 影像学
4. 在每一步后，用户输入自己的诊断假设
5. 最终揭示诊断 + 治疗 + 学习要点
6. 系统对比用户假设与实际诊断，给出评分

评分逻辑：
- 诊断正确 ✓
- 关键检查是否想到了
- 鉴别诊断是否全面
```

#### 2.2 后端新增命令

```
submit_case_attempt(case_id, user_diagnosis, user_tests) → score
get_case_study_stats(user_id) → { studied_count, accuracy }
```

#### 2.3 学习进度追踪

- 新增 `case_progress` 表：user_id / case_id / attempts / best_score / last_studied
- 在 CaseLibrary 卡片上显示"已学/未学"标记

**预估**：1 个新页面 + 2 条新命令 + 1 张新表 + 迁移

## 三、技术细节

### 数据库变更

```sql
-- Phase 2：学习进度
CREATE TABLE IF NOT EXISTS case_progress (
    user_id TEXT NOT NULL DEFAULT 'local',
    case_id TEXT NOT NULL REFERENCES cases(id),
    attempts INTEGER DEFAULT 0,
    best_score REAL DEFAULT 0,
    last_studied TEXT,
    PRIMARY KEY (user_id, case_id)
);

-- Phase 1：全文搜索
CREATE VIRTUAL TABLE IF NOT EXISTS cases_fts USING fts5(
    id, title, chief_complaint, diagnosis,
    content='cases',
    content_rowid='rowid'
);

-- 触发器保持同步
CREATE TRIGGER cases_ai AFTER INSERT ON cases BEGIN
    INSERT INTO cases_fts(id, title, chief_complaint, diagnosis)
    VALUES (new.id, new.title, new.chief_complaint, new.diagnosis);
END;
CREATE TRIGGER cases_ad AFTER DELETE ON cases BEGIN
    INSERT INTO cases_fts(cases_fts, rowid, id, title, chief_complaint, diagnosis)
    VALUES ('delete', old.rowid, old.id, old.title, old.chief_complaint, old.diagnosis);
END;
```

## 五、风险与注意事项

1. **数据一致性**：新增 `cases_fts` 触发器需与种子数据重建脚本联动
2. **CaseStudy 评分逻辑**：初步可只做字符串匹配，后续再用语义相似度
3. **Markdown 编辑器**：Phase 3 的表单需要 MD 编辑器，可复用 crossnote 或集成 simplemde
4. **迁移版本**：每次 DB 变更需递增 `SEED_DATA_VERSION`
5. **gen_from_yaml.py**：新增 `case_disease` 映射后需同步更新生成器中的硬编码映射
