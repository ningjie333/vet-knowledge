# 修复任务标准化流程（Fix SOP）

> 本文档定义 vet-knowledge 项目所有代码修复任务的标准化流程，确保修复过程可追溯、可验证、可回滚。
>
> **适用范围**：Bug 修复、技术债清理、代码质量改进、生产级质量提升等所有变更类任务。
>
> **设计原则**：三阶段闭环（推进 → 修复 → 评审），每阶段有明确入口与出口标准。

---

## 1. 修复任务分类

### 1.1 优先级定义

| 优先级 | 定义 | 响应时限 | 闭环要求 |
|--------|------|----------|----------|
| **P0** | 影响核心功能正确性或数据安全 | 立即 | 必须单元测试 + check.py --full |
| **P1** | 影响可靠性或存在潜在风险 | 1-3 天 | 必须单元测试 + check.py --quick |
| **P2** | 代码质量或可维护性改进 | 1-2 周 | check.py --quick |
| **P3** | 体验优化或长期演进 | 按计划 | 视情况 |

### 1.2 修复类型分类

| 类型 | 描述 | 典型场景 |
|------|------|----------|
| **Bug Fix** | 功能行为与预期不符 | 排序 panic、字段 undefined |
| **Debt Clear** | 已识别的技术债清理 | 空实现、硬编码、废弃资源 |
| **Quality Up** | 无 Bug 但需提升质量 | 补测试、补错误处理、补文档 |
| **Refactor** | 不改变外部行为的内部重构 | 提取公共函数、消除重复代码 |

---

## 2. 三阶段闭环流程

### 阶段一：推进（Plan）

**入口标准**：问题已识别，需进入修复流程。

**执行步骤**：

1. **根因分析**（强制）
   - 用 5-Why 或鱼骨图定位根本原因，禁止只治症状
   - 记录：问题现象 / 根因 / 影响范围 / 修复策略

2. **影响面评估**
   - 列出受影响文件清单
   - 评估是否影响数据契约（schema、API、类型定义）
   - 评估是否需要数据迁移

3. **修复方案设计**
   - 列出具体改动点（文件 + 函数 + 改动描述）
   - 标注是否破坏向后兼容
   - 如有多种方案，列出优劣对比

4. **测试用例设计**（P0/P1 强制）
   - 先写测试用例清单（输入/期望输出/验证点）
   - 覆盖正常路径 + 边界条件 + 错误路径

5. **TodoWrite 登记子任务**
   - 每个修复点拆为独立子任务
   - 标注优先级和依赖关系

**出口标准**：修复方案已确认，子任务已登记，可进入实施。

---

### 阶段二：修复（Implement）

**入口标准**：方案已确认，子任务已登记。

**执行步骤**：

1. **按方案实施改动**
   - 遵循现有代码风格（命令模式、Composition API、CSS 变量等）
   - 每个逻辑变更对应一次提交，禁止混合多个修复

2. **测试先行（P0/P1）**
   - 先写测试用例（应失败，证明 Bug 存在）
   - 再实施修复（测试应通过）
   - 禁止"先修复后补测试"

3. **错误处理遵循规范**
   - 参照 `_knowledge/workflows/error_handling_standard.md`
   - 禁止新增 `.ok()` 吞没、静默 `catch`、裸 `unwrap()`

4. **每完成一个子任务立即标记 TodoWrite**
   - 禁止批量完成多个任务后再统一标记
   - 标记完成时附带简短摘要（summary 字段）

**出口标准**：所有子任务完成，本地 `cargo test` 与 `cargo check` 通过。

---

### 阶段三：评审（Review）

**入口标准**：所有修复子任务已完成。

**执行步骤**：

1. **运行静态检查**
   ```bash
   python _tools/dev/check.py --config check_config.json --full
   ```
   - 退出码 0：通过，可进入下一步
   - 退出码 1（CONFIRMED）：必须修复后重新评审
   - 退出码 2（REVIEW_NEEDED）：人工判断是否可接受

2. **运行单元测试**
   ```bash
   cd src-tauri && cargo test
   ```
   - 所有测试必须通过
   - 新增测试覆盖率不低于该模块平均水平

3. **类型检查**
   ```bash
   pnpm vue-tsc --noEmit
   ```
   - 前端 TypeScript 必须零错误

4. **编译验证**
   ```bash
   cd src-tauri && cargo check
   pnpm build
   ```

5. **闭环确认**
   - 所有检查通过后，在 TodoWrite 中标记评审任务完成
   - 在 commit message 中标注闭环状态

**出口标准**：所有验证通过，修复任务闭环完成。

## 3. 修复任务登记模板

每个修复任务在 TodoWrite 中应包含以下信息（content 字段）：

```
【阶段X：分类】简要描述
```

示例：
```
【阶段二：P0 修复】engine.rs 诊断推理引擎补单元测试
【阶段二：P1 修复】迁移块用 IF NOT EXISTS 替代 .ok() 错误吞没
【阶段三：评审】运行 check.py --full 完成闭环验证
```

---

## 4. Commit Message 规范

### 4.1 格式

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 4.2 Type 清单

| Type | 描述 |
|------|------|
| `fix` | Bug 修复 |
| `refactor` | 重构（不改变外部行为） |
| `test` | 新增或修改测试 |
| `docs` | 文档变更 |
| `chore` | 构建/工具/配置变更 |
| `style` | 代码格式调整 |
| `perf` | 性能优化 |

### 4.3 Scope 清单

| Scope | 描述 |
|-------|------|
| `engine` | 诊断推理引擎 |
| `flashcards` | 闪卡系统 |
| `db` | 数据库层 |
| `commands` | Tauri 命令层 |
| `frontend` | 前端 Vue/TS |
| `data` | 数据管线 |
| `check` | 静态检查框架 |
| `docs` | 文档 |

### 4.4 示例

```
fix(engine): 修复排序时 NaN 导致的 panic

根因：partial_cmp().unwrap() 在 score 为 NaN 时 panic
修复：改用 unwrap_or(Ordering::Equal)
测试：新增 3 个边界用例覆盖 NaN/空输入/单元素

闭环：check.py --full 通过，cargo test 5/5 通过
```

## 5. 回滚策略

### 5.1 回滚触发条件

- 修复引入新 Bug 且无法快速修复
- 性能回退超过 20%
- 静态检查从通过变为失败
- 单元测试出现新的失败

### 5.2 回滚流程

1. `git revert <commit>` 回滚最近变更
2. 重新运行 `check.py --full` + `cargo test` 验证回滚后状态
3. 在 TodoWrite 中重新打开相关子任务
4. 重新进入阶段一（Plan）调整方案

## 6. 禁止行为清单

| 编号 | 禁止行为 | 理由 |
|------|----------|------|
| F-01 | 禁止只治症状不治根因 | 问题会复发 |
| F-02 | 禁止先修复后补测试 | 测试无法证明 Bug 存在 |
| F-03 | 禁止新增 `.ok()` 吞没错误 | 掩盖问题根因 |
| F-04 | 禁止新增静默 `catch` | 用户无反馈 |
| F-05 | 禁止裸 `unwrap()` 在生产代码 | panic 风险 |
| F-06 | 禁止混合多个修复在一个 commit | 无法独立回滚 |
| F-07 | 禁止跳过 check.py --full 闭环 | 质量无保障 |
| F-08 | 禁止修改 schema.sql 不递增迁移版本 | 数据不一致 |
| F-09 | 禁止修改数据源不重新生成 SQL | 数据库与源不同步 |
| F-10 | 禁止关闭检查器规避问题 | 治标不治本 |
