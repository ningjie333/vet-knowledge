## 一、总体架构

```
┌──────────────────────────────────────────────┐
│  Vue 3 前端 (症状选择 → 推理请求 → 结果展示)   │
│          invoke('infer_diagnosis', ...)        │
├──────────────────────────────────────────────┤
│  Tauri Command Layer (18 个注册命令)           │
├──────────────┬───────────────────────────────┤
│ 知识查询模块  │  推理引擎 (engine.rs)          │
│ (CRUD/搜索)  │  加权评分 + 双维度阈值过滤       │
├──────────────┴───────────────────────────────┤
│  sqlx 0.8 (async SQLite, 编译期 SQL 检查)     │
├──────────────────────────────────────────────┤
│  SQLite 数据库 (vet_knowledge.db)             │
└──────────────────────────────────────────────┘
```

**为什么选 Tauri 而非 Electron**：Rust 原生执行效率更高、内存占用更小，且 sqlx 提供编译期 SQL 检查，能在构建阶段拦截数据库查询错误——这对医学知识库的准确性至关重要。

- `diseases_fts`：搜索疾病名称 + 概述
- `symptoms_fts`：搜索症状名称 + 定义
- `drugs_fts`：搜索药物名称 + 分类

### 3.4 数据库初始化流程

1. `db::init()` 获取 `app_data_dir`（系统应用数据目录）
2. 创建 `vet_knowledge.db`，URI 格式：`sqlite://{path}?mode=rwc`（注意正斜杠替换反斜杠）
3. 执行 `schema.sql` 建表
4. 检查 `diseases` 表是否为空，若为空则执行 `001_initial.sql` 导入种子数据

## 五、推理引擎（核心算法）

**文件**：`src-tauri/src/engine.rs`

### 5.1 输入

```rust
struct DiagnosisInput {
    symptoms: Vec<String>,  // 用户选择的症状名称列表
    species: String,        // "犬" 或 "猫"
    age: Option<f64>,
    breed: Option<String>,
}
```

### 5.2 算法流程

对每种候选疾病计算：

1. **疾病覆盖度**（`disease_coverage`）= `weighted_score / total_weight`
   - 权重：common=1.0, uncommon=0.6, rare=0.3
   - 即该疾病的症状库中被用户症状匹配的加权比例

2. **输入覆盖度**（`input_coverage`）= `matched_count / input_count`
   - 即用户输入的症状中被该疾病覆盖的比例

3. **综合评分** = `disease_coverage × input_coverage`

### 5.3 阈值过滤

必须同时满足：
- 综合分 ≥ 0.25
- 输入覆盖度 ≥ 30%

### 5.4 为什么需要双维度

**历史问题**：原算法只有疾病覆盖度。用户选"体重下降 + 癫痫发作"时，某疾病只有"体重下降"一个 common 症状，会得到 100% 疾病覆盖度（因为唯一的症状被匹配了），错误地排在第一位。

**解决方案**：加入输入覆盖度后，该疾病最高只能得到 50% 输入覆盖度（匹配了 2 个中的 1 个），综合分被压在 0.5 以下。如果该疾病症状库较大，综合分更低，会被过滤掉。

### 5.5 输出

```rust
struct DiagnosisCandidate {
    disease_id: String,
    disease_name: String,
    match_score: f64,           // 综合评分（已取整到小数点后2位）
    input_coverage: f64,        // 输入覆盖度（百分比形式）
    matched_symptoms: Vec<String>,
    missing_key_symptoms: Vec<String>,  // 未匹配的 common 症状
    suggested_tests: Vec<TestSuggestion>,
    distinguishing_features: Vec<String>,  // 当前为空
}
```

结果按 `match_score` 降序排列，最多返回 10 个。

## 十二、已知问题与踩坑记录

### 已修复的问题

| 问题 | 原因 | 解决方案 |
|---|---|---|
| Rust 单引号字符串 | Rust 不支持单引号字符串字面量 | 改为双引号 |
| 缺少 engine/search 模块 | lib.rs 引用了未创建的模块 | 创建 engine.rs，移除对 search 模块的引用 |
| 数据文件路径错误 | `../data/` 实际应为 `../../data/` | 修正相对路径 |
| sqlx API 变更 | `&**pool` 在 sqlx 0.8 中不兼容 | 改为 `&*pool` |
| 缺少 Row trait | `r.get::<String, _>()` 需要 `use sqlx::Row` | 添加 import |
| 缺少 Manager trait | `app.manage()` 需要 `use tauri::Manager` | 添加到 db/mod.rs import |
| SQLite 无法打开 | Windows 路径反斜杠问题 | URI 格式 `sqlite://{path}?mode=rwc`，替换 `\` 为 `/` |
| 推理结果重复 | SQL JOIN 导致笛卡尔积 | 使用 `SELECT DISTINCT d.id, d.name_zh` |
| 推理结果不准确 | 单维度评分缺陷 | 改为双维度评分（疾病覆盖度 × 输入覆盖度）+ 阈值过滤 |

### 注意事项

1. **sqlx 编译期检查**：修改 SQL 查询后必须重新编译才能让 sqlx 验证 SQL 正确性
2. **种子数据修改**：修改 `tools/gen_sql*.py` 后必须重新运行 `run_gen.py` 才能更新 `001_initial.sql`
3. **数据库重置**：删除 `app_data_dir/vet_knowledge.db` 后重启应用会重新导入种子数据
4. **category 字段**：存储 JSON 数组字符串（如 `'["呼吸系统"]'`），前端需要 `JSON.parse()` 解析
5. **species 字段**：存储 JSON 数组字符串（如 `'["犬","猫"]'`），前端需要 `JSON.parse()` 解析
6. **etiology 字段**：存储 JSON 数组字符串，前端需要 `JSON.parse()` 解析
